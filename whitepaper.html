1. Executive summary

IBITIcoin (IBITI) is a modular DeFi token ecosystem deployed on BNB Smart Chain. The protocol includes a fee calculator, staking, NFT utility/discount layers, a DAO module, a buyback manager, sale/vesting modules, and a website purchase flow.

A key production feature is the Website Cash Promo + Referral Router deployed behind a verified UUPS proxy. It implements a non-custodial purchase flow: user pays USDT → USDT is forwarded to the reserve wallet immediately, while IBITI payouts (base + promo bonus + referral reward) are distributed from the router’s funded IBITI inventory.

This document focuses on verifiable mechanics: how prices are quoted, where funds go, what is upgradeable, what the owner can change, which parameters exist, and what risks remain.

UUPS proxy Upgradeable proxy pattern where implementation can be replaced by an authorized upgrade.
Pancake V2 pair IBITI/USDT liquidity pool used as on-chain price reference (reserves).

3. System overview & design principles

Design principles:

  • On-chain transparency: all core parameters live on-chain, verifiable via BscScan.
  • Modularity: independent contracts for fees, staking, NFT discounts, DAO, etc.
  • Non-custodial website purchase: payment token is forwarded immediately; router aims to keep zero payment balance.
  • Upgrade boundaries: most modules are non-upgradeable; the website router is upgradeable (UUPS) to patch critical issues.
  • Explicit operational wallets: reserve/funding actions are traceable transactions, not hidden accounting.

High-level architecture map:

User wallet
  ├─ trades on PancakeSwap (IBITI/USDT pair) ──► affects reserves and spot price
  └─ buys on website (Cash Promo Router)
        ├─ transfers USDT from user
        ├─ forwards USDT to Reserve Wallet (immediately)
        └─ transfers IBITI from Router inventory:
              base amount (quoted from pair reserves)
              + promo bonus (bonusBps)
              + optional referral reward (refReward)

5. IBITI token contract (IBITIcoin.sol)

The IBITI token contract is an ERC-20 with extended functionality (fees, staking/DAO hooks, freeze logic, and optional purchase utilities). It also inherits voting functionality (ERC20Votes) via the base token layer.

5.1 Supply, decimals, cap

Decimals 8 (constant).
Total supply cap 100,000,000 * 10^8 minted at deployment.
Mint destination Minted to founderWallet which is set to address(this) (token contract address).
Treasury freeze The contract “treasury alias” is frozen on outgoing transfers to prevent accidental drains.

Important nuance: in this system, the “treasury alias” (founderWallet) is the token contract itself. This is a defensive pattern: initial supply exists on-chain but is not meant to be casually moved via direct transfer.

5.2 Roles & operational separation

  • Owner: controls administrative settings (pausing, module addresses, fee toggles where applicable).
  • distributionWallet: external wallet where collected fees can be routed (set to the owner wallet in constructor).
  • reserveWallet: operational wallet for liquidity/sales/operations (separate from owner).
  • User status controls: freeze/blocked logic exists to mitigate abuse or critical incidents (must be used carefully).
Concrete mechanics — what reviewers typically check
  • decimals() is overridden to return 8 (not 18).
  • totalSupplyCap is an immutable deployment cap; minted once.
  • founderWallet = address(this) and is frozen for outgoing transfers after mint.
  • Token integrates modules: FeeManager, UserStatusManager, BridgeManager, StakingModule, DAOModule, and optional NFTDiscount.

5.3 ERC20Votes (governance readiness)

The base token layer includes ERC20Votes, enabling delegated voting power and snapshot-like vote tracking. This supports DAO modules that rely on token voting.

5.4 Fee flow (high-level)

Fees are calculated by FeeManager and applied under defined conditions. Fee proceeds are typically routed to the project distribution wallet (operational policy) or used for ecosystem funding. Exact routing depends on the token’s internal logic and configured flags.

6. Website Cash Promo + Referral Router (UUPS)

The website purchase contract is IBITIReferralPromoRouterCashUUPS.sol. Its purpose is not to replace PancakeSwap, but to implement a controlled on-chain sale UX with: promo bonus, referral reward, reserve-forwarding, and transparent limits.

6.1 Money flow (critical)

User (USDT) ── transferFrom ─► Router
Router ── transfer ─► Reserve Wallet  (immediately, same tx)
Router ── transfers IBITI ─► Buyer (base + bonus)
Router ── transfers IBITI ─► Referrer (+1 IBITI if eligible)

The router aims to hold no USDT after a purchase. IBITI is paid from router inventory funded by the project (bonus pool / inventory top-ups).

6.2 Core state parameters (on-chain)

promoActive Enables/disables purchases through the router.
promoEndTime Optional end timestamp. Purchases are rejected after end time.
bonusBps Bonus basis points (e.g. 1000 = 10%).
refReward Fixed IBITI amount for referral reward (e.g. 1 IBITI = 100000000 with 8 decimals).
minPaymentAmount / maxPaymentAmount Hard bounds per buy (USDT amount).
quoteMode 0 = AMM-style quote with fee; 1 = spot ratio (not recommended).
feeBps AMM fee basis points used in quoteMode 0 (Pancake V2 ~ 25 bps).
minSpotPrice / maxSpotPrice Optional price bounds against manipulation.
requirePauseForCritical Forces pause before critical config changes (safety policy).

6.3 Pricing and quoting logic (concrete)

The router reads reserves from the IBITI/USDT pair and computes a quote. There are two quote modes:

Quote mode 0 — AMM-style quote (recommended)

This mimics UniswapV2/PancakeV2 output formula using reserves and feeBps:

amountInWithFee = paymentAmount * (10000 - feeBps)
amountOut = (amountInWithFee * reserveOut) / (reserveIn*10000 + amountInWithFee)

This is a safer quote style because it naturally reflects slippage for larger buys.

Quote mode 1 — spot reserve ratio (not recommended)

Spot price is computed as payRes * 10^ibDec / ibRes, then output is paymentAmount * 10^ibDec / spotPrice. This ignores slippage and is generally less safe in volatile liquidity conditions.

6.4 Anti-manipulation features

  • Spot bounds: if minSpotPrice/maxSpotPrice are set, router rejects buys outside allowed range.
  • Slippage guard: caller provides minIbitiOut and tx reverts if quote is lower.
  • Buy limits: minPaymentAmount and maxPaymentAmount constrain per-tx exposure.

6.5 Promo payout math (exact)

  • Base amount: boughtAmount from quote
  • Bonus amount: (boughtAmount * bonusBps) / 10000
  • Referral reward: fixed refReward if referrer is valid and not already credited for this buyer/referrer pair
  • Pool accounting:
    • bonusPoolTotal — declared total pool size (for stats)
    • bonusSpent, refSpent — counters
    • poolRemaining() = remaining amount available for bonus+ref payouts

6.6 Key functions (for reviewers)

  • initialize(paymentToken, ibitiToken, pair, routerLike, reserveWallet, swapPath, bonusBps, refReward, minPayment, maxPayment, promoEndTime)
  • buyWithReferral(paymentAmount, referrer, minIbitiOut) — the purchase entrypoint
  • getSpotPrice(), quoteBoughtAmount(), quoteTotalToBuyer() — quoting helpers
  • Admin setters: setPromoParams, setPromoActive, setPromoEndTime, setBuyLimits, setQuoteConfig, setSpotPriceBounds, setReserveWallet, setPair, setTokens
  • Pool ops: fundBonusPool (approve-based), adjustBonusPoolTotal (sync stats), withdrawExtraIBITI, withdrawAllIBITI
  • Safety: pause(), unpause(), rescueToken()

6.7 Events (transparency)

  • PromoBuy(buyer, referrer, paymentAmount, boughtAmount, bonusAmount, refAmount)

7. Fee model (FeeManager)

FeeManager is a dedicated contract that calculates trading fees using base rates and modifiers. It is designed to keep fee logic auditable and configurable while clamping the result to safe bounds.

7.1 What FeeManager can account for

  • Base buy fee / base sell fee
  • Holding-time discounts (optional)
  • VIP discounts (optional)
  • Staking discounts (optional)
  • Whale / size-based surcharges
  • Volatility coefficient (tiered or auto-adjusted)
  • NFT discount integration (via NFTDiscount module)
Key functions — what’s typically reviewed
  • calculateFee(...) — computes the fee based on context and flags
  • updateActivity(...) — updates volume metrics for volatility logic (tokenContract gated)
  • Admin setters: setBaseFees, setMinFee, setMaxFee, setVolatilityParams, setVolatilityTiers, setHoldDiscountEnabled, setVipDiscountEnabled, setStakingDiscountEnabled
  • Safety: pause(), unpause(), auditParameters()

Note: FeeManager does not custody user funds. It is a calculation module called by the token contract, with parameters stored on-chain and observable.

8. Staking (StakingModule)

The staking module enables time-locked staking positions with reward rules defined on-chain. Staking may also integrate with fee discounts, depending on FeeManager configuration.

  • Users lock IBITI for a defined period.
  • Rewards are paid in IBITI (funded explicitly, not by infinite inflation).
  • Early exit can be penalized depending on configuration.

Reviewers should check: reward funding mechanism, lock enforcement, emergency withdraw policy, and owner controls.

9. NFTs & discounts (IBITINFT, NFTDiscount, NFTSaleManager)

NFTs provide utility/discount tiers in the IBITI ecosystem. Discount mapping is managed by NFTDiscount and can be consulted by FeeManager or website dApps.

9.1 IBITINFT

  • Official ERC-721 NFT collection contract.
  • Ownership can grant fee discounts or special access.

9.2 NFTDiscount

  • Maps NFT ownership and tiers to discount benefits.
  • Provides “use” logic for limited-use discount NFTs where applicable.

9.3 NFTSaleManager

Implements sale phases and pricing logic for NFTs. Reviewers should check sale caps, price calculation, and fund routing.

10. DAO governance (DAOModule)

The DAO module supports on-chain proposals and voting. It is intentionally scoped to minimize risk in early stages.

  • Proposals have a voting period and execution rules.
  • Execution should be restricted to safe/whitelisted actions (review policy).
  • Voting power can be based on token balance and/or delegated votes (ERC20Votes).

Governance is powerful and dangerous. The project keeps critical controls under owner authority until stable, and gradually expands DAO scope.

11. Buyback & burn (BuybackManager)

BuybackManager can acquire IBITI on PancakeSwap using stablecoins (operationally supplied) and burn a portion by sending to the burn address.

  • Buyback is optional and depends on treasury/reserve policy.
  • Swaps should use slippage protection and minimum-out checks.
  • Burn is transparent: tokens are sent to 0x...dEaD.

Buyback is not a profit guarantee; it is an operational tool that may or may not be used depending on market conditions.

12. Sale & vesting modules (PhasedTokenSale, TeamVesting)

These modules formalize scheduled releases and structured sale phases. Vesting exists to align long-term development incentives with transparent unlock rules.

  • PhasedTokenSale: structured phases, caps, and pricing rules.
  • TeamVesting: long-term vesting schedule with cliff/unlock controls.

13. Security model & upgrade boundaries

Security is primarily enforced by (a) strict fund flow design, (b) permission boundaries, (c) pause mechanisms, and (d) transparency.

13.1 Upgradeability

  • Upgradeable: Website Cash Promo Router (UUPS proxy) — to patch critical issues.
  • Non-upgradeable (typical): most ecosystem modules and the token are deployed as standard contracts.

13.2 Owner controls (what can be changed)

The owner can change parameters in modules that expose admin setters (e.g. promo bonus, limits, quote config, pause/unpause). This is visible on-chain and should be treated as a trust assumption until governance/multisig hardening is applied.

13.3 requirePauseForCritical

The promo router includes a safety policy flag requirePauseForCritical: when enabled, certain critical parameter changes can be required to happen only while paused, reducing attack surface during live traffic.

No smart contract is “risk free”. Bugs, market manipulation, liquidity issues, and operational mistakes can still lead to losses.

15. Risks & limitations (read this)

This section is intentionally blunt. It exists to protect users and to satisfy reviewer expectations.

15.1 Market & liquidity risk

  • Low liquidity can cause high slippage and volatile pricing.
  • Large trades can move reserves and affect quoted purchase amounts.

15.2 Pair reserve manipulation risk

  • Because the router uses pair reserves as a price reference, reserve manipulation is a known risk.
  • Mitigations: spot bounds, slippage checks (minIbitiOut), and buy limits.

15.3 Smart contract risk

  • Bugs or integration errors may cause loss of funds.
  • Upgradeability itself is a risk: owner-controlled upgrades are a trust assumption.

15.4 Operational risk

  • Reserve wallet management and liquidity operations must be handled carefully (allowances, approvals, router permissions).
  • Misconfiguration of promo limits or price bounds can break purchases or create unfair conditions.

Participation in DeFi is inherently risky. Use small amounts, verify contracts, and understand the flow before interacting.
